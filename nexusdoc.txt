Hosted repository: A hosted repository in Sonatype Nexus Repository Manager is used to store artifacts that are created within your organization. After Jenkins builds an application, artifacts such as JARs, WARs, or Docker images are uploaded here for versioning and future deployments. In short, it stores internally built packages.

Proxy repository: A proxy repository acts as an intermediary between your environment and external repositories such as Apache Maven Central. When developers request external dependencies, Nexus downloads them once from the internet, caches them locally, and serves future requests from cache. This improves build speed, reduces internet dependency, and gives better control over external packages.

Group repository: A group repository combines multiple hosted and proxy repositories into a single endpoint. Instead of configuring developers or CI pipelines with multiple repository URLs, they use one group repository URL, and Nexus automatically fetches artifacts from the appropriate underlying repository. It simplifies repository management and provides a single access point.

=====================================================================================================
Snapshot repository: A snapshot repository stores unstable or in-development versions of artifacts that are frequently updated during development. These versions usually have names like 1.0-SNAPSHOT, indicating that the build is not final and may change with every new commit or build triggered by Jenkins. It is mainly used for continuous integration and testing phases where developers need the latest build updates.

Release repository: A release repository stores stable, finalized versions of artifacts that are ready for production use. These versions follow fixed version numbers such as 1.0, 2.1, etc., and once uploaded, they typically should not be modified. After code passes validation steps like SonarQube Community Build checks and testing, the final artifact is pushed to the release repository for deployment.

#!/bin/bash

set -e

NEXUS_VERSION="3.79.1-04"
NEXUS_USER="nexus"
NEXUS_HOME="/opt/nexus"
NEXUS_WORK="/opt/sonatype-work"

echo "========== SYSTEM UPDATE =========="
sudo apt update -y

echo "========== INSTALL JAVA 21 =========="
sudo apt install -y openjdk-21-jdk wget tar unzip curl ufw

java -version

echo "========== CREATE NEXUS USER =========="
if id "$NEXUS_USER" &>/dev/null; then
    echo "User exists"
else
    sudo useradd -M -d $NEXUS_HOME -s /bin/bash $NEXUS_USER
fi

echo "========== CLEAN OLD SETUP =========="
sudo systemctl stop nexus || true
sudo rm -rf $NEXUS_HOME
sudo rm -rf /tmp/nexus*

echo "========== CREATE DIRECTORIES =========="
sudo mkdir -p $NEXUS_HOME
sudo mkdir -p $NEXUS_WORK

echo "========== DOWNLOAD NEXUS =========="
cd /tmp
wget https://download.sonatype.com/nexus/3/nexus-${NEXUS_VERSION}-linux-x86_64.tar.gz

echo "========== EXTRACT =========="
tar -xvzf nexus-${NEXUS_VERSION}-linux-x86_64.tar.gz

echo "========== MOVE FILES =========="
sudo mv nexus-${NEXUS_VERSION}/* $NEXUS_HOME/

echo "========== SET PERMISSIONS =========="
sudo chown -R $NEXUS_USER:$NEXUS_USER $NEXUS_HOME
sudo chown -R $NEXUS_USER:$NEXUS_USER $NEXUS_WORK

echo "========== CONFIGURE RUN USER =========="
echo "run_as_user=\"$NEXUS_USER\"" | sudo tee $NEXUS_HOME/bin/nexus.rc

echo "========== SYSTEMD SERVICE =========="
sudo tee /etc/systemd/system/nexus.service > /dev/null <<EOF
[Unit]
Description=Nexus Repository Manager
After=network.target

[Service]
Type=forking
LimitNOFILE=65536
User=$NEXUS_USER
Group=$NEXUS_USER
ExecStart=$NEXUS_HOME/bin/nexus start
ExecStop=$NEXUS_HOME/bin/nexus stop
Restart=on-abort

[Install]
WantedBy=multi-user.target
EOF

echo "========== OPEN PORT =========="
sudo ufw allow 8081/tcp || true
sudo ufw --force enable || true

echo "========== START NEXUS =========="
sudo systemctl daemon-reload
sudo systemctl enable nexus
sudo systemctl restart nexus

echo "Waiting for startup..."
sleep 60

echo "========== STATUS =========="
sudo systemctl status nexus --no-pager

SERVER_IP=$(curl -s ifconfig.me)

echo ""
echo "=============================================="
echo " Nexus Installed Successfully"
echo " URL: http://$SERVER_IP:8081"
echo ""
echo " Admin Password:"
echo " sudo cat /opt/sonatype-work/nexus3/admin.password"
echo "=============================================="

pipeline script:


pipeline {
    agent any
    tools {
       maven 'maven.15'
    }

    environment {
        NEXUS_USER = 'admin'
        NEXUS_PASS = 'admin'
        NEXUS_REPO_URL = 'http://13.201.36.207:8081/repository/maven-release-repo'
        GROUP_ID = 'com.evolve'
        ARTIFACT_ID = 'evolve-technologies'
        VERSION = '1.0'
        PACKAGING = 'war'
        FINAL_NAME = 'evolve-technologies'
    }

    stages {
        stage('Checkout Code') {
            steps {
                git url: 'https://github.com/umamahesh571/repo-name', branch: 'main'
            }
        }

        stage('Build WAR') {
            steps {
                sh 'mvn clean package -DskipTests'
            }
        }

        stage('Upload WAR to Nexus') {
            steps {
                sh '''
                    FILE_PATH=target/${FINAL_NAME}.${PACKAGING}
                    GROUP_PATH=$(echo ${GROUP_ID} | tr '.' '/')
                    UPLOAD_URL=${NEXUS_REPO_URL}/${GROUP_PATH}/${ARTIFACT_ID}/${VERSION}/${ARTIFACT_ID}-${VERSION}.${PACKAGING}
                    
                    echo "Uploading WAR to: $UPLOAD_URL"
                    
                    curl -v -u ${NEXUS_USER}:${NEXUS_PASS} --upload-file $FILE_PATH $UPLOAD_URL
                '''
            }
        }
    }

    post {
        success {
            echo '✅ WAR file built and uploaded to Nexus successfully.'
        }
        failure {
            echo '❌ Failed to build or upload the WAR file to Nexus.'
        }
    }
}

ROLES OF DEVOPS ENGINEER IN NEXUS:

As a DevOps engineer, your role with Sonatype Nexus Repository Manager is usually around managing artifacts and integrating Nexus into CI/CD pipelines, not developing application code.

Repository Management: You create and manage repositories such as hosted, proxy, and group repositories based on project requirements. For example, creating snapshot and release repositories for internal artifacts and proxy repositories for external dependencies.

CI/CD Integration: You integrate Jenkins pipelines with Nexus so that once the build is completed, artifacts like JARs, WARs, or Docker images are automatically pushed to Nexus.

Artifact Versioning: You ensure proper version control of artifacts by managing snapshot and release versions, helping teams track builds and perform rollbacks when needed.

Access Management: You configure users, roles, and permissions so that only authorized teams can upload, download, or manage artifacts.

Dependency Optimization: You configure proxy repositories to cache dependencies from external sources like Apache Maven Central, reducing build time and minimizing internet dependency.

Storage Maintenance: You monitor disk utilization, clean unused artifacts, configure retention policies, and maintain repository health.

Backup and Recovery: You ensure Nexus data is backed up regularly so artifacts are not lost during failures.

Security & Compliance: You may integrate Nexus with vulnerability scanning tools and ensure only approved artifacts are used in deployments.

Deployment Support: During deployments, you ensure teams pull the correct artifact version from Nexus instead of rebuilding the application again.

In short, as a DevOps engineer, Nexus becomes your central artifact management system in the CI/CD pipeline.
